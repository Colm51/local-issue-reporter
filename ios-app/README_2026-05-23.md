# LocalIssueReporter

LocalIssueReporter is a beginner-friendly SwiftUI iPhone app for creating simple local issue reports and sending them through the native iOS Mail composer. It is a prototype, not an official municipal reporting app.

The app is intentionally simple. It does not use a backend, cloud sync, accounts, a database server, or third-party iOS packages.

## What The App Does

LocalIssueReporter helps a user prepare a local issue report with:

- an issue type
- optional notes
- the captured latitude and longitude
- an Apple Maps link with a visible pin label that includes the report ID
- a unique report ID
- a timestamp
- the destination email address
- up to 3 photo attachments
- an automatic per-report CSV attachment

The report is sent using Apple's native Mail compose sheet. The app does not send email directly by itself.

## Current App Features

### Native Mail Report Submission

Reports are prepared inside the app and then handed to `MFMailComposeViewController`, Apple's built-in Mail composer. The user can review and edit the email before sending.

When Mail reports that the message was sent, the app saves the report to local submitted history and returns to the main screen. A temporary success message appears on the main screen.

Cancelled messages, saved drafts in Mail, and failed sends do not get saved as submitted reports.

### Default Destination Email

The report form pre-fills a default destination email address. The user can edit the address before sending.

### Photos

The report form supports multiple photos:

- up to 3 photos per report
- camera or photo library selection
- small previews in the form
- selected photos attached to the Mail message as JPEG files

### Location Capture

The app uses Core Location to capture the current device location. The captured latitude and longitude are included in:

- the email body
- the per-report CSV attachment
- the submitted report history
- the Apple Maps link

The location authorization flow is handled through the Core Location authorization callback to avoid main-thread authorization warnings.

### Report IDs

Each report gets a unique short report ID. The report ID is included in:

- the email body
- the per-report CSV attachment
- the submitted report history
- the submitted reports list
- the submitted report detail screen
- the Apple Maps pin label
- the per-report CSV filename

Per-report CSV attachments use a filename like:

```text
local-issue-report-ABC123.csv
```

### Automatic CSV Attachment

Every submitted email includes a CSV file with one row for the report.

The CSV columns are:

```text
report_id, issue_type, notes, latitude, longitude, apple_maps_link, timestamp, destination_email, photo_count
```

The existing email body and photo attachments are kept.

### Submitted Reports History

The app saves a local submitted report record after Mail returns `.sent`.

The history stores:

- report ID
- issue type
- notes
- latitude
- longitude
- Apple Maps link
- timestamp
- destination email
- photo count

The Submitted Reports screen shows a list of saved reports. Tapping a report opens a detail screen with the full saved values.

The history can also be:

- cleared after confirmation
- exported as one CSV file using the native iOS share sheet

The exported all-reports CSV uses a dated filename such as:

```text
submitted-reports-2026-05-23.csv
```

### Persistent Draft Support

The app supports one saved draft report.

A draft preserves:

- issue type
- notes
- destination email
- captured latitude and longitude
- selected photos

Draft data is stored locally. The draft record is encoded as JSON in `UserDefaults`, while draft photo files are saved separately in the app's Documents directory under `DraftPhotos/`.

Draft photos are resized before saving. The longest edge is limited to about 1600 pixels, and the files are saved as JPEG with compression around 0.75. The app stores only photo filenames in the draft record, not raw image data.

When a draft is continued, the app restores the saved text fields, destination email, captured location, and draft photos. If some draft photos cannot be restored, the app continues loading the draft and shows a friendly message.

When a draft is submitted successfully or discarded, the draft JSON and draft photo files are deleted.

### Custom App Icon

The app includes a custom minimalist bicycle icon in the asset catalog so the app has a recognizable home screen icon.

## Project Structure

### Swift App Files

`LocalIssueReporterApp.swift`

The app entry point. It launches `ContentView`.

`ContentView.swift`

The main SwiftUI view. It manages the main screen, report form, selected photos, report creation, Mail completion handling, submitted report saving, success messages, and draft actions.

`LocationManager.swift`

Handles Core Location authorization and current location updates.

`MailComposeView.swift`

Wraps `MFMailComposeViewController` so the SwiftUI app can present the native Mail composer.

`ReportEmail.swift`

Represents the email content and attachments prepared for a report.

`SubmittedReport.swift`

Defines submitted report records, local submitted report storage, CSV export support, and the SwiftUI submitted reports screens.

`DraftReport.swift`

Defines the one saved draft report record. The draft stores text values, destination email, captured coordinates, saved draft photo filenames, and the save timestamp.

`Assets.xcassets`

Contains app visual assets, including the custom bicycle app icon.

### Documentation Files

The project includes original and dated documentation files. Dated files preserve snapshots of the project state over time.

### Separate Python Mapping Workflow

The Python mapping workflow lives outside the iOS app project in a separate `Reports` folder and separate Git repository.

The mapping script:

- reads exported LocalIssueReporter CSV files
- combines them into one pandas DataFrame
- preserves `report_id` as text
- converts latitude and longitude to numeric values
- drops rows without valid coordinates
- saves a cleaned combined CSV
- generates an interactive Folium HTML map
- colors report markers by issue type
- shows report details in hover tooltips and marker popups
- adds a legend titled `Reports from Local Issue Reporter`
- adds a bike lane and multi-use trail overlay from a GeoPackage file
- includes a separate bike lanes legend
- writes dated and versioned map and CSV outputs

Example generated filenames:

```text
local_issue_report_map_2026-05-23_v1.html
combined_reports_cleaned_2026-05-23_v1.csv
```

The mapping workflow uses Python packages such as pandas, Folium, GeoPandas, and Branca. It is a local analysis tool, not part of the iOS app runtime.

## Storage And Privacy

LocalIssueReporter stores data only on the device unless the user chooses to send or share it.

Local storage includes:

- submitted reports saved in `UserDefaults`
- one draft report saved in `UserDefaults`
- draft photo JPEG files saved in the app Documents directory

The app does not provide:

- user accounts
- cloud sync
- backend storage
- server-side processing
- automatic municipal system integration

Users should avoid including sensitive personal information in report notes or photos.

## Limitations

This is a prototype. Current limitations include:

- reports are sent through Mail, not directly into a municipal ticketing system
- submitted history is local to the device
- draft support is limited to one saved draft
- draft photos are resized copies, not original full-resolution images
- there is no duplicate detection
- there is no moderation, triage workflow, or official service request tracking
- map generation is a separate local Python workflow

Possible future enhancements could include direct integration with an official intake API, accessibility improvements, multilingual support, duplicate detection, and administrative triage tools.
