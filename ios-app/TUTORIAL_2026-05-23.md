# LocalIssueReporter Tutorial

This tutorial explains how to run the current LocalIssueReporter prototype, create reports, save drafts, view submitted history, and use the separate Python mapping workflow.

LocalIssueReporter is intentionally simple. It uses SwiftUI, native iOS frameworks, local storage, and the native Mail composer. It does not use a backend, cloud sync, accounts, or third-party iOS packages.

## 1. Open And Run The App

1. Open the project in Xcode.
2. Select an iPhone simulator or a physical iPhone.
3. Build and run the app.
4. If testing Mail sending, use a device or simulator setup that can present the native Mail composer.

The app opens to the main screen with actions for creating a new report and viewing submitted reports. If a saved draft exists, the main screen also shows draft actions.

## 2. Create A New Report

1. Tap `New Report`.
2. Choose an issue type.
3. Enter notes if needed.
4. Review or edit the destination email address.
5. Tap the location refresh action to capture the current location.
6. Add photos if needed.
7. Send the report when ready.

Reports can include up to 3 photos. The selected photos appear as small previews before sending.

## 3. Save A Draft

Use Save Draft when you want to start a report at the issue location and finish it later.

1. Open the report form.
2. Choose an issue type.
3. Add notes if needed.
4. Capture the current location.
5. Add photos if needed.
6. Tap `Save Draft`.

A draft requires a captured location. If no location has been captured, the app shows the existing validation alert.

When a draft is saved:

- one draft record is saved locally
- selected photos are saved as resized JPEG files
- the report form closes
- the main screen shows `Draft Saved`

Only one draft is supported. Saving a new draft replaces the previous one.

## 4. Continue A Draft

If a draft exists, the main screen shows `Continue Draft`.

1. Tap `Continue Draft`.
2. The app restores the saved issue type, notes, destination email, location, and photos.
3. Continue editing the report.
4. Save it again as a draft or submit it through Mail.

If some saved draft photos cannot be restored, the app still opens the draft and shows a message.

## 5. Discard A Draft

If a draft exists, the main screen shows `Discard Draft`.

Tap it to delete:

- the saved draft record
- saved draft photo files

Discarding a draft does not affect submitted report history.

## 6. Submit A Report

When the report is ready:

1. Tap the send action in the report form.
2. Review the native Mail compose sheet.
3. Send the email.

The app keeps the existing Mail flow. The user can still edit the email before sending.

When Mail reports that the message was sent:

- the app saves the report to submitted history
- the app deletes any saved draft and draft photo files
- the app returns to the main screen
- the app shows `Report Successfully Submitted` temporarily

Cancelled messages, saved drafts in Mail, and failed messages are not saved as submitted reports.

## 7. What Gets Sent

Each report email includes:

- an email body with the report details
- selected photo attachments
- a per-report CSV attachment

The per-report CSV filename includes the report ID:

```text
local-issue-report-ABC123.csv
```

The CSV columns are:

```text
report_id, issue_type, notes, latitude, longitude, apple_maps_link, timestamp, destination_email, photo_count
```

The Apple Maps link includes a query label and coordinates so Maps shows a visible pin for the report location.

## 8. View Submitted Reports

1. Return to the main screen.
2. Tap `Submitted Reports`.
3. Review the list of locally saved submitted reports.
4. Tap a report to see full details.

The list shows key values such as report ID, issue type, timestamp, and coordinates.

## 9. Clear Submitted Reports

The Submitted Reports screen includes an option to delete saved submitted reports.

The app asks for confirmation first. Confirming clears the local submitted report history from `UserDefaults`.

This does not affect emails that were already sent.

## 10. Export Submitted Reports As CSV

The Submitted Reports screen includes an `Export All as CSV` action.

The app creates one combined CSV file with all locally saved submitted reports and opens the native iOS share sheet. From there, the file can be shared by Mail, AirDrop, Files, or another available share destination.

The exported filename includes the current date, for example:

```text
submitted-reports-2026-05-23.csv
```

If there are no submitted reports, the app shows a friendly message instead of opening an empty share sheet.

## 11. Run The Python Mapping Workflow

The mapping workflow is separate from the iOS app. It is intended for local analysis of exported CSV files.

In the separate `Reports` folder:

1. Put exported LocalIssueReporter CSV files in the folder.
2. Activate the local virtual environment if available:

```bash
source reports_env/bin/activate
```

3. Install required packages if needed:

```bash
python3 -m pip install pandas folium geopandas branca
```

4. Run the mapping script:

```bash
python3 create_local_issue_report_map.py
```

The script reads LocalIssueReporter CSV files, combines and cleans the report rows, then creates:

- a cleaned combined CSV
- an interactive Folium HTML map

Generated output filenames are dated and versioned:

```text
combined_reports_cleaned_2026-05-23_v1.csv
local_issue_report_map_2026-05-23_v1.html
```

If version 1 already exists for the date, the script creates version 2, then version 3, and so on.

## 12. Mapping Features

The generated map includes:

- report markers
- marker colors by issue type
- hover tooltips with report ID and issue type
- popups with report ID, issue type, timestamp, notes, and photo count
- a report legend titled `Reports from Local Issue Reporter`
- a bike lane and multi-use trail overlay
- a separate bike lanes legend
- layer controls so the bike lane layer can be toggled

The bike lane overlay uses a local GeoPackage file. If the file is missing or cannot be read, the script prints a warning and still generates the report map.

## 13. Basic Git Workflow

Before making changes:

```bash
git status --short
```

Review changes:

```bash
git diff
```

Stage intended files:

```bash
git add path/to/file
```

Commit:

```bash
git commit -m "Describe the change"
```

The iOS app project and the separate `Reports` mapping folder may be separate Git repositories. Check the current folder before running Git commands.

Generated mapping files and virtual environment files should stay ignored in the mapping repository, including generated HTML maps, generated CSV files, `reports_env/`, and `__pycache__/`.

## 14. Privacy Reminder

Reports may include location coordinates, notes, photos, and email addresses. Treat exported CSV files and generated maps as potentially sensitive.

Do not share report exports or maps publicly unless the contents have been reviewed.
